# PROPOSAL: Project AWRA Web Dashboard (GAS Edition)

## 1. Design Concept: "The Operational Command Center"
Instead of a standard weather report website, we design this as a **Command Center Dashboard**. The goal is instant situational awareness for WFM (Workforce Management) and Operations Managers.

### **Core Identity**
*   **Theme**: Dark Mode with "Glassmorphism" (semi-transparent cards).
*   **Palette**: 
    *   Dark Slate Background (Professional)
    *   **Neon Green**: Normal Operations
    *   **Caution Amber**: Monitoring Required (Rain/Traffic risk)
    *   **Alert Red**: Critical Action Needed (Typhoon/Flooding)

### **Proposed Layout (Wireframe)**
1.  **Header (The "Go/No-Go" Signal)**
    *   Full-width ticker tape at the top showing active systems (e.g., *"NO ACTIVE TROPICAL CYCLONE"*).
    *   Massive Status Indicator: **"OPERATIONAL STATUS: NORMAL"**
2.  **Site Cards (The "Executive Summary")**
    *   Two prominent cards side-by-side: **Metro Manila** and **Dumaguete** (Visayas).
    *   **Contents**:
        *   Big Weather Icon (Sun/Cloud/Rain).
        *   **Executive Summary Text**: The concise, 1-sentence summary generated by our AI (e.g., *"Expect light rains, no transport impact valid for 24h"*).
        *   **Key Stats**: Temp | Wind | Rain % (Small pills).
3.  **The Impact Matrix**
    *   A consolidated table showing risk levels for:
        *   **Commute/Transport**
        *   **Power/Internet**
        *   **Health/Safety**
4.  **10-Day Horizon**
    *   A collapsible accordion or horizontal scroll view for the long-term AccuWeather data.

## 2. Technical Implementation (GAS + Python) Since the front-end will be GAS.

Since your heavy lifting (PAGASA Scraping, PDF Parsing, OpenAI Analysis) requires Python, we will use a **Hybrid Architecture**.

### **The Workflow**
1.  **Python Backend (Your existing `run_pipeline.py`)**:
    *   Runs locally (or on a scheduled server).
    *   Generates the `summary.json` (now with `_email` and `_detailed` fields).
    *   **New Step**: Pushes this JSON to a specific **Google Sheet** (acting as your database) or overwrites a JSON file in **Google Drive**.
2.  **GAS Frontend (The Web App)**:
    *   `Code.gs`: Fetches the data from the Sheet/Drive file `doGet()`.
    *   `Index.html`: Renders the beautiful dashboard using the data.

### **Why this Approach?**
*   **Easy Deployment**: You just share the `script.google.com` URL with your team. No servers to manage for them.
*   **Security**: It runs within your corporate Google Workspace.
*   **Power**: You keep the power of Python/Playwright (which GAS cannot do easily) while getting the deployment ease of GAS.

## 3. Recommended Tech Stack for GAS Frontend
*   **Tailwind CSS (via CDN)**: For rapid, beautiful styling without writing 1000 lines of CSS.
*   **Vue.js (via CDN - optional)**: To make the dashboard "reactive" (update instantly without reloading), or just plain Vanilla JS for simplicity.
*   **Google Sheets as DB**: Easiest way to debug. You can manually tweak the data in the sheet if the Python script makes a mistake, and the dashboard updates instantly.
